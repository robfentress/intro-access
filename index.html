    Web Content Accessibility Guidelines

    • 4 Design Principles:
      Perceivable, Operable, Understandable, and Robust
      • Contain 12 Guidelines
        • Testable success criteria
          • Each criteria at level A, AA, AAA
    1. Read: The Web Content Accessibility Guidelines are organized by four design principles, Perceivable, Operable, Understandable, and Robust, which form the nice acronym, POUR. Breaking that out means that:
      • Information and user interface components must be presentable to users in ways they can perceive;
      • User interface components and navigation must be operable;
      • Information and operation of the user interface must be understandable; and
      • Content must be robust enough that it can be interpreted reliably by a wide variety of user agents, including assistive technologies.
    2. Reveal next fragment, Contain 12 Guidelines
    3. Read: Under each Principle, there are Guidelines.
    4. Reveal next fragment, Testable success criteria
    5. Read: Under each Guideline, in turn, there are a number of specific, testable success criteria.
    6. Reveal next fragment, on Criteria
    7. Read: Each success criteria is assigned a conformance level, A, AA, or AAA. If your web page meets all of the success criteria that are listed as AA, you can claim WCAG AA conformance for that page.

Web Accessibility Initiative – Accessible Rich Internet Applications, which you'll usually just hear called ARIA, is a technical specification published by the World Wide Web Consortium (W3C) that describes how to increase the accessibility of web pages that have more application-like functionality.     Link text should describe purpose

    Avoid "click here" or "read more".

    1. Read: Because screen reader users can navigate by jumping from link to link using a key command, it is important that the text of any links describes their purpose. For instance, where the link takes the user or the name of the document that would be downloaded. This saves these users time by letting them quickly navigate to where they need to go.
    2. Reveal the first fragment
    3. Read: However, if the links contain more generic text, such as "click here", especially if there are multiple "click here" links, this affordance is less useful, as we can see from the list of links being presented to users of the macOS screen reading software, VoiceOver. Where would I go if I clicked on one of these links?
    4. Reveal the next fragment
    5. Read: Rather than saying "click here" or "read more", it is better to structure your content so the link text describes the destination.

      • One other way to magnify things that doesn't require separate software is to use your browsers native zooming functionality.
      • For instance in Safari I can magnify the page by pressing Command + + or choosing Zoom In from the View menu.
      • This will make the text bigger, but can cause problems with horizontal scrolling unless you are using responsive design techniques that transform the layout as the viewport size decreases, which is essentially what is happening when you use Browser Zoom.

      • Some people may not want to zoom the entire page, and may prefer to just make the text on the page bigger. For instance, you can set your preferred text size in Chrome's preferences window.

      • However, unlike browser zoom, this technique only works if the page has been coded to use relative units for font size. These relative units include em, rem, %, and keywords like large.
      • If, instead, you have used absolute measures, such as pixels, points, or inches, they will not change when the user sets their preferred text size in their browser's preferences.
      • Regardless, the key thing to keep in mind here is that users should be able to set their text to be up to 200% of its normal size one way or another, without any of the text becoming truncated or overlapping other content and thus becoming unreadable.
